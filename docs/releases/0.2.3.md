# mutsu 0.2.3 — release & store notes

## What's new (user-facing changelog)

**Watching on sites that can't be embedded (Netflix-style) is much smoother**

- When a source won't load in the room, you now get a clear **"Watch in your own
  tab"** action instead of a bare browser "refused to connect" page — and while
  it's checking, a tidy "Loading…" cover explains what's happening.
- If you already have that site open, mutsu **pairs your existing tab
  automatically** — no extra "Open tab" click.
- New: **"Play this page for everyone."** Browse to a different title on a
  streaming site, then push it to the whole room with one click (respects the
  room's control mode); everyone's tab follows.

**In-tab widget**

- Cleaner header: the **room name is a link** that jumps to the room tab, plus
  dedicated **minimize** (to a floating bubble) and **close** buttons.
- Closing shows a brief hint on how to reopen it (from the toolbar popup).

**Popup**

- The widget show/hide controls and live status appear **instantly** instead of
  waiting for the page scan.
- A page that plays its own video ("page with player") is now offered as a
  **watch-in-your-own-tab** source rather than a frame that would be refused.

**Fixes**

- "Reopen tab" now **focuses the existing tab** instead of opening duplicates.
- The in-tab widget is now properly **removed when the room switches** to a
  different (embeddable) source.

## Notes for store reviewers

### What the extension does
mutsu keeps a group's **video playback in sync** for a watch party. Everyone
plays their own copy of the same source in their own browser; the extension reads
and nudges the `<video>` element's **clock** (play / pause / seek / current time)
so playback stays aligned. It does **not** download, record, relay, re-encode, or
unlock any video — no media bytes ever pass through any server. It only moves a
URL and a timestamp.

A companion web page (the "room", https://mutsu.onesal.me) holds the group
connection; the extension's job is purely to script the `<video>` in pages the
web page itself can't reach (cross-origin iframes, or a separate streaming tab).

### Data collection & privacy
- **No data collection. No analytics, telemetry, tracking, ads, or cookies.**
- The only thing sent off the device is the chosen source **URL** and the
  **playback position**, to the room server, so peers can sync. No personal data,
  no browsing history, no page content, no video.
- Firefox: declared as `data_collection_permissions: { required: ["none"] }`.
- No accounts; a nickname is stored in `localStorage` on the web page only.

### No remote code
- All logic ships in the package. The extension loads **no remote scripts** and
  uses no `eval`/remote code execution. (It connects only to the room server for
  sync and to fetch user-requested subtitles/GIFs via that server.)

### Permission justifications
- **`host_permissions: <all_urls>` + content script on all frames** — the source
  a group watches can be *any* site/embed provider chosen at runtime, so the
  content script must be able to run in any page/iframe to find and sync its
  `<video>`. It stays inert on pages that aren't part of an active party (it only
  engages after the room page hands it a sync session).
- **`scripting`** — the toolbar popup scans the *active* tab for `<video>` /
  `<iframe>` sources so the user can pick what to watch together.
- **`tabs`** — for "watch this page in its own tab" parties, a background service
  worker relays sync messages between the user's room tab and their streaming tab
  (the only context that can message between two tabs) and notices when either
  closes. Also used to find the open room tab to deliver a picked source.
- **`storage`** — holds the local tab-pairing state for the above (which room tab
  drives which streaming tab). Session-scoped; cleared on browser close.
- **`web_accessible_resources: icon/*.png`** — the in-tab widget's floating
  bubble shows the extension icon, which a content-script `<img>` can only load if
  the icon is web-accessible.

### How to test
1. Open https://mutsu.onesal.me — it creates a room (you can also open the URL in
   two browser profiles to simulate two people).
2. Click the mutsu toolbar button → "New room", or paste a video/embed URL in
   the room's **Source** panel.
3. Play/pause/seek in one window and observe the other staying in sync.
4. For "watch in your own tab": pick a site that blocks embedding; the room shows
   "Watch in your own tab", which opens/pairs a tab the extension then syncs.

No login or paid account is required to review the core sync behavior.

### Firefox (AMO) specifics
- Build: Manifest V2 (`.output/firefox-mv2`), add-on id `sixseven@onesal.me`.
- The submission includes a **sources zip** (`*-sources.zip`) alongside the build
  zip, since the build output is minified by the bundler.

#### Reproducing the build (for AMO source review)
- Environment: Node.js ≥ 22, pnpm 9.15.0 (see `packageManager` in the root
  `package.json`), any OS (built/tested on macOS).
- Steps from the repository root:
  ```bash
  pnpm install
  pnpm --filter @mutsu/extension build:firefox   # → .output/firefox-mv2
  # (or `… zip -b firefox` to reproduce the submitted zips)
  ```
- The submitted artifact corresponds to `packages/extension` at version 0.2.3.
  No build-time secrets are required; the room server host is a public URL baked
  into the build (`mutsu.onesal.me`), not a credential.
